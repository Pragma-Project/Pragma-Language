Memory Safety

What it is, why entire industries spend millions on it, and why ~70% of critical vulnerabilities come from getting it wrong.

What Memory Safety Is

Every program uses memory — variables, strings, arrays, objects, everything lives in memory while the program runs. Memory safety means the program only accesses memory it's supposed to access, in the way it's supposed to access it, for the duration it's supposed to access it.

An unsafe program can read memory that belongs to something else (buffer overread), write past the end of an array (buffer overflow), use memory after it's been freed (use-after-free), or free the same memory twice (double-free). These aren't just bugs that crash your program. They're the primary attack surface for security exploits.

The Bugs That Break Everything

Buffer overflow

You allocate an array of 10 elements and write to index 15. The write lands on whatever happens to be next to your array in memory — maybe another variable, maybe a return address. An attacker who controls the input can use this to execute arbitrary code. This is how the Morris Worm worked in 1988. It's still how exploits work today.

Use-after-free

You free a block of memory, then continue using the pointer. The memory may have been reallocated to something else. You're now reading or writing someone else's data. This class of bug is responsible for a significant portion of Chrome and Firefox CVEs every year.

Null pointer dereference

You access memory through a pointer that was never initialized or was set to null. The program crashes or, on some systems, the attacker can map memory at address zero and control what the program reads.

Data races

Two threads access the same memory at the same time, and at least one is writing. The result depends on timing — which thread happens to run first. The program works 99% of the time and fails catastrophically the other 1%, in ways that are nearly impossible to reproduce or debug.

Who Cares About This

Everyone who builds software that can't afford to fail:

Aviation. Airbus and Boeing follow DO-178C, a certification standard that requires demonstrating absence of certain classes of errors in flight-critical software. A memory bug in an autopilot system is not a crash report — it's an actual crash. Airbus has invested heavily in formal verification tools to prove memory safety mathematically rather than relying on testing alone.

Automotive. ISO 26262 governs functional safety in vehicles. With modern cars running 100+ million lines of code across dozens of ECUs, memory corruption in brake control or steering software is a safety-of-life concern. MISRA-C exists specifically to constrain how C is written in these contexts.

Medical devices. The FDA requires software validation for any device that touches patient care. A buffer overflow in an insulin pump or pacemaker firmware is a direct threat to human life. IEC 62304 defines the software lifecycle requirements.

Finance. Trading systems, payment processors, and banking infrastructure handle billions of dollars in transactions. A memory corruption bug that alters a transaction amount or allows unauthorized access has immediate, measurable financial impact.

Operating systems and browsers. Google has reported that roughly 70% of serious Chrome security bugs are memory safety issues. Microsoft has reported the same ratio for Windows CVEs. These are not obscure edge cases — they're the dominant source of critical vulnerabilities in the software that billions of people use daily.

Industry Standards

MISRA-C

MISRA-C is a set of coding guidelines for C, originally developed for the automotive industry by the Motor Industry Software Reliability Association. It restricts which C features you can use — no dynamic memory allocation, no recursion, no pointer arithmetic in certain contexts. The goal is to eliminate the patterns that lead to memory bugs. It doesn't prove safety; it reduces the probability of unsafe code by banning dangerous constructs. It's widely adopted in automotive, aerospace, rail, and medical device development.

Formal verification

Rather than testing a program with a finite set of inputs, formal verification uses mathematical proof to show that a property holds for all possible inputs. Tools like Frama-C (for C) and SPARK (for Ada) can prove that a program never accesses out-of-bounds memory, never dereferences a null pointer, and never has a data race. This is the gold standard — not "we tested it and it didn't fail" but "we proved it cannot fail." The cost is that writing the proof annotations and running the solvers takes significant effort.

How Languages Handle It

C. Doesn't handle it. C trusts the programmer completely. You can do anything — cast any pointer, write past any buffer, free memory and keep using the pointer. Tools like Valgrind and AddressSanitizer catch some of these at runtime, but only for the inputs you happen to test.

Java, Python, JavaScript. Prevent most memory bugs by using garbage collection and hiding raw pointers. You can't have a buffer overflow in Python because the runtime checks every array access. The cost is performance and the loss of low-level control.

Rust. Prevents memory bugs at compile time through the borrow checker — an ownership system that tracks which code owns which memory and prevents you from using it incorrectly. It works, but it's mandatory. Every line of Rust code is subject to the borrow checker whether you need it or not. You can opt out with unsafe blocks, but then you're writing those annotations everywhere and losing the ergonomics that were supposed to be the point. Learning lifetimes, ownership, and borrowing has a real cost, and you don't get to skip it for the parts of your program where safety doesn't matter.

How UniLogic Handles It

Unlike Rust, memory safety in UL is an option, not a requirement. If you're writing a quick script or prototyping, you don't have to think about it at all — defaults apply and you move on. When you need safety guarantees, you turn them on per file with a DR directive (safety: checked), and the compiler proves memory safety mathematically. It generates proof obligations and an SMT solver discharges them. If every obligation is satisfied, the program is proven safe for every possible input — not just the ones you tested. No lifetime annotations. No fighting a borrow checker.

This is the same class of guarantee that Airbus uses for flight-critical software and that MISRA-C attempts to approximate through coding restrictions. UL builds it into the compiler toolchain — but only when you ask for it.

Proven, not tested Testing shows the absence of bugs for the inputs you tried. Proof shows the absence of bugs for all inputs that could ever exist. UL's MemProof system targets the latter.
