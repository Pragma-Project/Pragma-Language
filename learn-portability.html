Deployment & Platform Dependence

Native binaries (C, Rust, Go) · Virtual machines (Python, Java) · Browser runtime (WebAssembly) — why each breaks, and how UL gives you all three.

What Platform Dependence Is

Software doesn't exist in a vacuum. It runs on a specific operating system (Windows, Linux, macOS), a specific CPU architecture (x86-64, ARM), with specific libraries installed at specific versions. Change any of these and the software may not run at all.

This is not an edge case. It's the default state of affairs for most software. A program compiled on Ubuntu 22.04 may not run on Ubuntu 20.04 because the glibc version differs. A macOS app built for Apple Silicon won't run on an older Intel Mac. A Python script that works with Python 3.11 may fail on 3.9 because a function was added in 3.10.

The industry has developed three fundamentally different strategies to deal with this. Each solves part of the problem and creates new problems of its own.

Strategy 1: Compile to Native — The C Approach

C, C++, Rust, and Go compile source code directly to machine code for a specific CPU and OS. The result is a standalone binary — no interpreter, no VM, no runtime dependency. It runs at full hardware speed because there's nothing between the program and the CPU.

This is the fastest possible execution model. Operating system kernels, game engines, databases, and embedded systems are written this way because they need every cycle the hardware can give them.

The cost: you build once per target. A Linux x86-64 binary is useless on Windows. A Windows ARM binary won't run on an Intel Mac. If your users are on 3 operating systems and 2 architectures, you maintain 6 separate builds, each with its own CI pipeline, its own testing matrix, and its own set of platform-specific bugs. Cross-compilation toolchains exist but are complex to set up and maintain.

Linux and macOS users are frequently left out when a developer only builds for Windows. Open source partially solves this by letting users compile from source — but that requires them to have the right compiler, the right system headers, and every transitive dependency installed correctly. Most users can't or won't do this.

Strategy 2: Run on a VM — The Python/Java Approach

Python, Java, C#, and Ruby take a different approach. Instead of compiling to native code, they compile to an intermediate representation (bytecode) that runs on a virtual machine. The VM is the platform-specific part — you install it once per platform, and then any program targeting that VM runs on it.

Java's "write once, run anywhere" promise relies on this. You compile your Java code to bytecode, and the JVM on each platform handles the rest. Python distributes source directly and the interpreter compiles it to bytecode on the fly.

This solves the portability problem well. But it introduces new ones:

Performance. There's an interpreter or JIT compiler between your code and the hardware. For Python, this means 10-100x slower execution than C for CPU-bound work. Java's JIT narrows the gap significantly but still can't match ahead-of-time compilation in all cases, and startup time suffers.

Dependency hell. Python's package ecosystem is a cautionary tale. pip install pulls in a tree of dependencies, each pinned to different version ranges, which may conflict with each other or with system packages. Virtual environments, conda environments, Docker containers — all exist primarily because managing Python dependencies on a bare system is unreliable. "Works on my machine" is not a solved problem.

Runtime requirement. The user must have the correct version of the runtime installed. A Python 3.12 script won't run if the user only has Python 3.8. A Java 21 app won't run on JRE 11. Distributing the runtime with your app (e.g., bundling a JRE) increases the download size significantly.

Strategy 3: Run in the Browser — The WebAssembly Approach

WebAssembly (Wasm) compiles code to a portable binary format that runs inside any modern browser. No installation, no platform-specific builds, no runtime to manage. Open a URL, and the application runs. This is true platform independence.

Wasm is fast — it compiles to near-native speed because browsers have highly optimized Wasm runtimes. It's sandboxed by default, which is good for security but limits what the application can do. File system access is restricted. Network access goes through browser APIs. Hardware access (GPU, USB, serial ports) is limited or unavailable.

WASI (WebAssembly System Interface) is extending Wasm beyond the browser, but the ecosystem is still young. For full applications that need OS-level access, Wasm alone isn't enough yet.

The Tradeoff Triangle

The three strategies form a triangle. Pick any two:

Native speed + full OS access — but no portability (compiled languages).

Portability + ecosystem — but slower and dependency-fragile (VMs/interpreters).

Universal access + no install — but sandboxed and limited (Wasm/browser).

No mainstream language today gives you all three from the same source code.

How UniLogic Solves This

UL compiles the same source to all three targets:

C and LLVM IR — native binaries. Full speed, full OS access, direct hardware control. For systems programming, embedded targets, and performance-critical applications.
