Deployment & Portability

Why getting software to run on someone else's machine is still one of the hardest problems in computing.

The Problem

You write software on your machine. It works. You send it to someone else. It doesn't work. This has been true since the 1960s and remains true today. The reasons come down to platform dependence — software is built for a specific operating system, CPU architecture, and set of installed libraries. Change any of those and things break.

"Works on my machine" is not a joke. It's the single most common failure mode in software deployment. An application that runs perfectly in development may crash on the first line in production because a library version differs, a path doesn't exist, or the CPU doesn't support an instruction the compiler assumed was there.

Compiled Languages: Native Speed, Platform Lock-in

C, C++, Rust, and Go compile to native machine code. The compiler produces a binary that runs directly on the CPU — no interpreter, no runtime overhead. This is as fast as software gets.

The cost: you need a separate binary for every platform. A Linux x86-64 binary won't run on macOS ARM. A Windows .exe is useless on Linux. Cross-compilation exists but is notoriously painful — different system headers, different linkers, different ABIs. If you target 3 operating systems and 2 CPU architectures, you're maintaining 6 build pipelines.

Linux and macOS users routinely can't use software that was only built for Windows. Open source projects mitigate this by providing source code, but that pushes the compilation burden onto the user — who now needs the right compiler, the right version, and every dependency installed correctly.

Interpreted Languages: Portable, But Constrained

Python, Ruby, and JavaScript take the opposite approach. You distribute source code (or bytecode), and an interpreter or VM on the user's machine runs it. Write once, run anywhere — as long as the interpreter is installed.

Python is the clearest example. Your script runs on any platform with a Python interpreter. But the tradeoffs are real:

Performance. Interpreted code is 10-100x slower than compiled code for compute-heavy work. CPython executes one bytecode instruction at a time. JIT compilers (PyPy, V8 for JavaScript) help, but can't match ahead-of-time compilation.

The GIL. CPython's Global Interpreter Lock means only one thread executes Python code at a time. True CPU parallelism requires multiprocessing, which is heavier and more complex.

Dependency hell. Python's package ecosystem is enormous but fragile. pip install pulls in transitive dependencies that may conflict with each other or with system packages. Virtual environments help but add complexity. Anyone who's debugged a broken numpy install on a fresh machine knows the pain.

"Works on my machine." Even with interpreted languages, differences in installed packages, Python versions, OS-level libraries, and environment variables cause deployments to fail. Docker exists largely because this problem is so pervasive — it ships the entire OS environment along with the application.

WebAssembly: Universal, But Limited

WebAssembly (Wasm) compiles code to a portable bytecode format that runs in any browser. No installation, no platform-specific builds — open a URL and the application runs. This is genuinely platform-independent.

But Wasm has its own constraints. It still compiles at load time (from bytecode to native code). It's sandboxed with limited access to the filesystem, network, and system resources. WASI (WebAssembly System Interface) is expanding these capabilities, but it's still early — you can't do everything a native application can. And for anything beyond the browser, the tooling is still maturing.

How UniLogic Handles This

UL doesn't pick one of these approaches. It offers all three through its multi-target compilation system:

Native binaries. XPile compiles UL to C or LLVM IR, which then compiles to native machine code. Full speed, direct hardware access, no runtime overhead. For systems work, embedded targets, and performance-critical applications.

Interpreted targets. XPile compiles UL to Python or JavaScript. The output runs anywhere those interpreters exist. For scripting, prototyping, and environments where portability matters more than raw speed.

Web deployment. The JavaScript target runs in any browser. Combined with the REPL that already runs client-side in the browser with zero dependencies, UL code can be deployed as a web application that anyone can access without installing anything.

One source, three deployment models The same .ul file compiles to a native binary, a Python script, or a browser application. The developer chooses the target at compile time, not at language-design time. No rewrites, no porting.
