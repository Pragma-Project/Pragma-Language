What is UniLogic?

Q: What is this called?

UniLogic. One syntax that covers the range from low-level systems work to high-level application development. Runtime behavior that is declared rather than inherited from whichever language you happened to pick. A schema-based expression layer that describes what a program does in the same terms you would use to explain it to another person, rather than the terms required to build it in an IDE.

This is a hobby project and an ongoing design exercise, not a product. The work is documented as it develops. Feedback and collaboration from people who find this worth thinking about is genuinely welcome.

Q: What's some differences in runtime behaviors?

Rust already toggles safety with unsafe blocks. Inside unsafe, you get raw pointers, manual memory, no borrow checker. Outside, you get full safety guarantees. Same language, two modes, toggled per-block.

C# has unsafe context too, by letting you use pointers and fixed-size buffers, opting out of GC for that scope.

D has @nogc annotation, which marks functions that aren't allowed to allocate from the garbage collector. You can mix GC and manual memory in the same program.

Zig lets you choose allocators per-call. You can have arena allocation in one place, general purpose style allocation in another, with no allocation at all in some hot path. You pass the allocator explicitly.

Swift has both ARC (reference counting) and unsafe pointers available in the same language.

No language puts all of these on a single dial panel. Every one picks a default and lets you escape from it in limited ways. Here, you declare your policy and the compiler enforces it.

Every function, data structure, and IO operation works under all configurations. A linked list under GC doesn't need a destructor. Under manual memory it does. Under refcount it needs weak reference support. These are the sorts of things that are a coverage goal, and is a necessary part of a system of unified syntax.

One standard library with an implementation of everything is the route to achieving a successful build for this concept. There's no novel ideas and reinventing the wheel, but rather just patching together and recreating past work. However, there is some design of interfacing and other systems that allow for co-existence of these systems.

The syntax decisions borrow from what already exists across C, Python, Java, and Javascript. The angle of attack in this component is to unify and minimize, not to invent. I want anyone to be able to see it and understand it. Verbosity is chosen over efficiency or what many might see as elegance. It's not something prioritizing functional programming; the entire personality and image being built is one revolving around practicality.

The full design rationale and conceptual basis is documented in the LANG Project document.
