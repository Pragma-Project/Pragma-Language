Memory Safety

This learning module covers the concepts behind memory safety — why it matters, where the danger comes from, and what different languages have done to address it. These ideas are essential background for understanding how UL approaches safety, and why that approach differs from every other language.

We start from the ground floor. What is memory? What happens when a program runs? How does C give you direct access to it, and why does that access create danger? Then: what Java, Python, and others did to combat that danger — and what they gave up in return. Finally, what Rust tried, and what UL does differently.

No prior knowledge of systems programming, hardware, or compilers is assumed. Each concept builds on the previous one.

Section 2 of 20

What UL Does Differently

Every language that came before UL picked one answer to memory safety and forced it on every program. C picked "the programmer handles it." Java and Python picked "the runtime handles it, and you pay the performance cost." Rust picked "the compiler handles it, and you fight the borrow checker on every line."

UL makes it a per-file configuration. One file in your project can use garbage collection. Another can use manual memory. A third can use reference counting. A fourth can use arena allocation. Each file declares its own memory model, and the compiler checks that they compose safely across file boundaries.

When you need the strongest guarantee — mathematical proof that memory bugs cannot exist — you set safety: checked and the compiler's MemProof system proves it. No lifetime annotations. No borrow checker. No runtime overhead. Just a proof obligation discharged by an SMT solver.

But to understand why this matters, you first need to understand where the danger comes from.

Section 4 of 20

What Memory Is

When a program runs, the operating system gives it a block of memory — a flat, numbered array of bytes. Each byte has an address, starting from zero and counting up. On a 64-bit system, addresses are 64 bits wide, which means the address space is enormous — far larger than physical RAM.

Everything your program works with — variables, strings, arrays, objects, function return addresses — lives somewhere in this flat array. An int might occupy bytes 1000 through 1003. A string might occupy bytes 2048 through 2060. They're just numbers at addresses.

The operating system manages which addresses are valid. If your program tries to read or write an address it wasn't given, the OS kills it — that's a segmentation fault. But within the addresses your program does own, the program is free to read and write anything, in any order, at any time. The OS doesn't check whether you're reading your own int or accidentally reading the return address of the function that called you.

This freedom is the root of the entire problem.

The Stack

Programs are made of functions. main() calls compute(), which calls format(), which calls strlen(). Each function needs its own space for local variables and parameters. The call stack is the region of memory that provides this space.

Every time a function is called, a stack frame is pushed onto the top of the call stack. That frame holds the function's local variables, its parameters, and a return address — the exact instruction to jump back to when the function finishes. When the function returns, its frame is popped off and the memory is instantly reclaimed. No cleanup needed.

This is why local variables in C are "free" from a performance perspective: allocation is a single pointer adjustment, and deallocation is automatic when the function returns.

      
But the stack has a critical limitation: everything on it has a fixed lifetime tied to the function call. You can't return a pointer to a local variable — the frame it lived in is gone the moment the function returns. For data that needs to outlive the function that created it, you need the heap.

    


    
    
    
    

      
Section 6 of 20

      
The Heap

      
The heap is a region of memory where you can allocate blocks of any size, at any time, and they persist until you explicitly free them. In C, you request heap memory with malloc() and release it with free().

      
Unlike the stack, heap memory has no automatic lifetime. It doesn't disappear when the function returns. This is its power — and its danger. If you forget to call free(), the memory leaks. If you call free() and then keep using the pointer, you have a use-after-free. If you call free() twice, you have a double-free.

Section 7 of 20

      
Pointers: Power and Danger

      
A pointer is a variable that holds a memory address. In C, pointers are the primary mechanism for working with memory. You can point to anything — an integer, a string, a function, another pointer. You can do arithmetic on pointers: add offsets, subtract them, cast them to different types.

      
This gives C its power. A pointer to the beginning of an array, plus an offset, gives you any element. A pointer to a struct member lets you modify data in place without copying. A function pointer lets you call any function dynamically. The entire Linux kernel is built on pointer manipulation.

      
int arr[5] = {10, 20, 30, 40, 50};
int *p = arr;         // p points to arr[0]
printf("%d", *(p+2));  // prints 30
printf("%d", *(p+7));  // reads PAST the array — undefined behavior

      
But the compiler doesn't know the array is 5 elements long. It doesn't stop you from reading index 7, or 100, or −3. The pointer is just a number. Arithmetic on it is just math. Where that number points — whether it's your data, someone else's data, or a return address — is entirely your responsibility.

      
This is not a theoretical concern. It is the mechanism behind the majority of security exploits in the history of computing.

Buffer Overflow

      
You allocate an array of 10 elements and write to index 15. The write doesn't fail — it lands on whatever happens to be next to your array in memory. Maybe another variable. Maybe the function's return address.

An attacker who controls the input can use a buffer overflow to overwrite the return address with a pointer to their own code. When the function returns, instead of going back to the caller, it jumps to the attacker's payload. This is how the Morris Worm worked in 1988. It is still how exploits work today.

      

        Real-world impact
        Google has reported that roughly 70% of serious Chrome security bugs are memory safety issues. Microsoft has reported the same ratio for Windows CVEs. Buffer overflows are the dominant source of critical vulnerabilities in the software that billions of people use daily.
      

    


    
    
    
    

      
Section 9 of 20

      
Use-After-Free & Double-Free

      
You call free(ptr). The memory is returned to the allocator. But your pointer variable still holds the old address — it's now a dangling pointer. If you read through it, you get whatever the allocator has done with that memory since. If you write through it, you corrupt whatever now lives there.

      
char *name = malloc(16);
strcpy(name, "Alice");
free(name);
// name still holds the old address
printf("%s", name);  // use-after-free: reads freed memory
free(name);          // double-free: corrupts allocator metadata

      
A double-free is worse. The allocator's internal data structures (free lists, chunk headers) are metadata stored in the heap itself. Freeing the same block twice corrupts those structures. A subsequent malloc() may return a pointer to memory that's already in use by something else, silently creating overlapping allocations. This is exploitable.

      
These bugs are particularly dangerous because they're intermittent. The program may work correctly 999 times out of 1000. The one failure depends on timing, allocator state, and what else the program has done with memory. They are extremely difficult to reproduce and debug.

    


    
    
    
    

      
Section 10 of 20

      
Null Pointers & Data Races

      
Null pointer dereference

      
You access memory through a pointer that was never initialized or was set to null. The program crashes — or, on some systems, an attacker can map memory at address zero and control what the program reads. Tony Hoare, who invented the null reference in 1965, famously called it his "billion-dollar mistake."

      
Data races

      
Two threads access the same memory at the same time, and at least one is writing. The result depends on which thread runs first — pure timing. The program works 99% of the time and fails catastrophically the other 1%, in ways that are nearly impossible to reproduce.

      
// Thread A            // Thread B
counter++;            counter++;
// Both read counter=5, both write 6
// Expected: 7. Actual: 6. Silent corruption.

      
Data races don't crash. They silently produce wrong results. In a financial system, this means incorrect transaction amounts. In a medical device, it means incorrect dosages. In a self-driving car, it means incorrect sensor readings.

      

        The common thread
        Every one of these bugs — buffer overflow, use-after-free, double-free, null dereference, data race — exists because C gives the programmer direct, unchecked access to memory and the compiler does not verify that access is correct. The language trusts the programmer completely. The question every subsequent language has tried to answer is: how do you remove that trust without removing the power?
      

    


    
    
    
    

      
Section 11 of 20

      
The Cost of C's Freedom

      
C was designed in 1972 for building Unix. Memory was scarce, CPUs were slow, and every byte and cycle mattered. The language was designed to give programmers complete control over the machine. No runtime checks. No garbage collector. No safety net.

      
This design was correct for 1972. But the world kept using it for the next 50 years, for applications its designers never imagined: web servers, browsers, medical devices, financial systems, self-driving cars. Code that faces hostile input from the internet. Code where a single bug can compromise millions of machines.

      
The tools that exist to help — Valgrind, AddressSanitizer, static analyzers — catch bugs at test time, for the specific inputs you happen to test. They do not prove absence of bugs. A program that passes every test can still have a buffer overflow that only triggers on a specific malicious input.

      
Industry responded with coding standards like MISRA-C (Motor Industry Software Reliability Association), which bans dangerous C features — no dynamic allocation, no recursion, restricted pointer arithmetic. It reduces the probability of unsafe code by eliminating dangerous patterns. But it's a restriction, not a proof. And it makes C even harder to write.

      
What was needed was a fundamentally different approach to memory management. Two competing philosophies emerged.

    


    
    
    
    

      
Section 12 of 20

      
Garbage Collection: The First Solution

      
The idea is simple: instead of making the programmer free memory, have the runtime do it automatically. A garbage collector (GC) periodically scans all memory, finds objects that are no longer reachable from any variable, and frees them. The programmer never calls free(). Dangling pointers can't exist because the GC only frees objects no one references.

      
Java (1995) was the first mainstream language to make GC mandatory and hide raw pointers entirely. You allocate with new and the GC handles deallocation. Array accesses are bounds-checked at runtime — writing past the end throws an exception instead of silently corrupting memory. Null pointer access throws NullPointerException instead of crashing.

      
Python (1991) uses reference counting plus a cycle-detecting GC. Every object has a counter tracking how many references point to it. When the count drops to zero, the object is freed immediately. A separate collector handles circular references.

      
JavaScript (1995), Go (2009), C# (2000) — all GC languages. The pattern was clear: the industry moved decisively away from manual memory management.

      
Garbage collection eliminated entire classes of bugs overnight. No more buffer overflows. No more use-after-free. No more double-free. The programmer could focus on logic instead of memory bookkeeping.

      
But it came at a cost.

GC pauses. When the collector runs, it must scan the entire heap to determine what's alive. During this scan, the program may be paused — Java's GC can pause for tens of milliseconds, sometimes hundreds. For a video game, that's a visible stutter. For a trading system, that's missed trades. For a real-time control system, that could be catastrophic.

      
Memory overhead. GC languages typically use 2–5x more memory than equivalent C code. The GC needs headroom to work efficiently — it performs best when the heap is less than half full. Objects carry metadata (headers, type tags, reference counts). Every allocation goes through the GC's bookkeeping.

      
No low-level control. You can't control when memory is freed, how it's laid out, or where it's allocated. You can't use arena allocation for a batch of short-lived objects. You can't place data in specific memory regions for cache efficiency. The GC decides.

      
This is why performance-critical code is still written in C and C++. NumPy, PyTorch, TensorFlow, Chrome's rendering engine, the Linux kernel, game engines — all C/C++ underneath. Python is the interface; C is the engine.

    


    
    
    
    

      
Section 14 of 20

      
Reference Counting

      
Reference counting is a middle ground between manual management and full garbage collection. Every object tracks how many pointers reference it. When a new pointer is created, the count increments. When a pointer is destroyed or reassigned, the count decrements. When the count hits zero, the object is freed immediately — no GC pause, no scan.

      
obj *a = new_obj();   // refcount = 1
obj *b = a;            // refcount = 2
b = NULL;              // refcount = 1
a = NULL;              // refcount = 0 → freed immediately

      
Advantages: Deterministic. Objects are freed the instant they're no longer needed — no pauses, predictable performance. This is why Python uses it (with a supplementary cycle collector) and why Apple chose it for Swift (ARC — Automatic Reference Counting).

      
Disadvantages: Every pointer assignment requires incrementing and decrementing counters. In tight loops, this overhead adds up. And reference counting cannot handle cycles: if object A references B and B references A, both refcounts stay at 1 forever, even if nothing else references them. Python's supplementary cycle detector handles this, but it's an extra pass.

      
Reference counting is a real option — deterministic, low-latency, good for real-time systems. But it's still a form of automatic management that imposes overhead on every pointer operation.

    


    
    
    
    

      
Section 15 of 20

      
Rust's Approach: The Borrow Checker

      
Rust (2015) tried a third approach: prevent memory bugs at compile time through an ownership and borrowing system. Every value has exactly one owner. When the owner goes out of scope, the value is dropped. You can lend a reference (borrow) but the compiler enforces rules: you can have either one mutable reference or any number of immutable references, never both.

      
let s1 = String::from("hello");
let s2 = s1;     // s1 is MOVED to s2 — s1 is now invalid
println!("{}", s1); // compile ERROR: value used after move

      
This eliminates use-after-free, double-free, and data races at compile time — no runtime cost, no GC. It's genuinely effective.

      
The cost is developer experience. The borrow checker is mandatory for all code. Every function, every struct, every reference must satisfy the ownership rules. This means fighting the compiler on patterns that are correct but that it can't prove safe. Self-referential structs, graph structures, and many common patterns require unsafe blocks, reference-counted wrappers (Rc, Arc), or redesigning your data structures to satisfy the borrow checker.

      
You don't get to turn it off for the parts of your program where safety doesn't matter. Writing a quick script? Prototyping? The borrow checker is still there. The learning curve is steep, and the productivity cost is constant.

      

        The core tension
        Rust proved that compile-time memory safety is achievable without garbage collection. But it also proved that making it mandatory creates friction that never goes away. The question becomes: is there a way to get the same guarantee without the constant cost?
      

    


    
    
    
    

      
Section 16 of 20

      
Who Needs Memory Safety

      
Everyone who builds software that can't afford to fail:

      
Aviation. Airbus and Boeing follow DO-178C, a certification standard that requires demonstrating absence of certain error classes in flight-critical software. A memory bug in an autopilot system is not a crash report — it's an actual crash. Airbus has invested heavily in formal verification tools to prove memory safety mathematically.

      
Automotive. ISO 26262 governs functional safety in vehicles. Modern cars run 100+ million lines of code across dozens of ECUs. Memory corruption in brake or steering software is a safety-of-life concern. MISRA-C exists specifically to constrain how C is written in these contexts.

      
Medical devices. The FDA requires software validation for any device that touches patient care. A buffer overflow in an insulin pump or pacemaker firmware is a direct threat to human life. IEC 62304 defines the software lifecycle requirements.

      
Finance. Trading systems, payment processors, and banking infrastructure handle billions in transactions. A memory corruption bug that alters a transaction amount or allows unauthorized access has immediate financial impact.

      
Browsers and operating systems. Google: ~70% of serious Chrome vulnerabilities are memory safety bugs. Microsoft: same ratio for Windows CVEs. Mozilla: same for Firefox. These are the dominant attack surface for software used by billions.

    


    
    
    
    

      
Section 17 of 20

      
Formal Verification

      
There's a fourth approach, used in the highest-stakes environments: mathematical proof. Rather than testing a program with some inputs and hoping, formal verification proves that a property holds for all possible inputs.

      
An SMT solver (Satisfiability Modulo Theories) takes a mathematical description of your program and tries to find an input that violates a property — for example, "array index is within bounds." If it can't find one, and it's searched exhaustively, the property is proven. Not "tested." Proven.

      
Tools like Frama-C (for C) and SPARK (for Ada) can prove that a program never accesses out-of-bounds memory, never dereferences null, and never has a data race. This is the gold standard: not "we tested it and it didn't fail" but "we proved it cannot fail."

      
The traditional cost is enormous. Writing proof annotations is laborious. Running solvers is slow. The tooling is academic and fragile. Typically only the most critical code — flight control, nuclear reactor controllers, spacecraft — justifies the effort.

      
UL's insight: what if the compiler could generate those proof obligations automatically, without requiring the programmer to write annotations?

How UL Handles It

      
UL's memory model is a per-file configuration, declared through a DR (Dynamic Runtime) directive at the top of the file:

      
// file: network.ul
@dr memory = gc
@dr safety = checked

// file: kernel_driver.ul
@dr memory = manual
@dr safety = checked

// file: quick_script.ul
// (defaults apply — gc, checked)

      
Four memory models:

      
gc — garbage collected, like Java. Safest, most convenient, highest overhead.

      
refcount — reference counted, like Swift. Deterministic, no pauses, per-operation overhead.

      
arena — arena allocation. All objects in a batch are freed together. Perfect for request-handling, parsers, game frames.

      
manual — malloc/free, like C. Full control, full responsibility.

      
When safety: checked is set, the compiler's MemProof system generates proof obligations and an SMT solver (CVC5) discharges them. If every obligation is satisfied, the program is mathematically proven safe for every possible input. No lifetime annotations. No borrow checker. No runtime overhead. Just a proof.

      
The compiler also detects cross-file conflicts — if one file uses GC and another uses manual memory, and they share data, the compiler flags the mismatch before the program runs.

      

        Proven, not tested
        Testing shows the absence of bugs for the inputs you tried. Proof shows the absence of bugs for all inputs that could ever exist. UL's MemProof system targets the latter — the same class of guarantee that Airbus uses for flight-critical software.
